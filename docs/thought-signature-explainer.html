thought_signature Bug

Why the agentic loop failed with 400 Bad Request

Background — what is a thought_signature?

gemini-3.5-flash has thinking mode on by default. Before answering, the model privately reasons through the problem — like mental scratch work.

Whenever the model decides to call a function while thinking, the Gemini API stamps the function call with a thought_signature — an opaque token that ties the function call back to the specific reasoning turn that produced it.

On follow-up requests, the API expects that thought_signature to still be present in the conversation history. If it's missing → 400 Bad Request.

What the SDK sends — before vs what Gemini expects

What SDK sends (broken)
{ "contents": [ { "role": "user", "parts": [{ "text": "I go to school..." }] }, { "role": "model", "parts": [{ "functionCall": { "name": "save_english_mistake", "args": { ... } // thought_signature missing! } }] }, { "role": "user", "parts": [{ "functionResponse": { "name": "save_english_mistake", "response": { "result": "saved" } } }] } ] }
What Gemini expects (correct)
{ "contents": [ { "role": "user", "parts": [{ "text": "I go to school..." }] }, { "role": "model", "parts": [{ "functionCall": { "name": "save_english_mistake", "args": { ... }, "thought_signature": "AaBbCc..." // ← must be preserved! } }] }, { "role": "user", "parts": [{ "functionResponse": { "name": "save_english_mistake", "response": { "result": "saved" } } }] } ] }

The fix — hybrid approach

Keep chat.sendMessageStream for the first turn (works fine — no history manipulation). For the follow-up turn, bypass ChatSession entirely and call model.generateContentStream with a manually-built history that includes the raw stream parts — thought_signature intact.
Before — chat.sendMessageStream for follow-up
// Turn 1 — fine const r1 = await chat .sendMessageStream(userMessage) // SDK stores chunks in history // → thought_signature is dropped here // Turn 2 — history is already broken const r2 = await chat .sendMessageStream([ { functionResponse: ... } ]) // → 400 Bad Request
After — model.generateContentStream for follow-up
// Turn 1 — same as before const r1 = await chat .sendMessageStream(userMessage) // Capture raw parts ourselves during streaming // → thought_signature is preserved in rawModelParts const rawModelParts: unknown[] = [] for await (const chunk of r1.stream) { for (const part of (chunk as any).candidates ?.[0]?.content?.parts ?? []) { rawModelParts.push(part) // raw → keeps sig } } // Turn 2 — bypass ChatSession, build history manually const manualHistory = [ ...previousMessages, { role: 'user', parts: [{ text: userMessage }] }, { role: 'model', parts: rawModelParts }, // ← sig preserved! { role: 'user', parts: [{ functionResponse: ... }] }, ] const r2 = await model .generateContentStream( { contents: manualHistory } as any ) // → Gemini sees thought_signature → 200 OK

Key insight

The SDK's ChatSession is the only broken piece. model.generateContentStream() — a lower-level method on the same SDK — just serialises whatever object you pass it directly to JSON and sends it to the API. It has no history management, so nothing gets stripped.

The fix uses ChatSession only for the first turn (where it works fine), then drops to the lower level for the follow-up where precise history control matters.
