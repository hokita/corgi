thought_signature Bug

Why the agentic loop failed with 400 Bad Request

Background — what is a thought_signature?

gemini-3.5-flash has thinking mode on by default. Before answering, the model privately reasons through the problem — like mental scratch work.

Whenever the model decides to call a function while thinking, the Gemini API stamps the function call with a thought_signature — an opaque token that ties the function call back to the specific reasoning turn that produced it.

On follow-up requests, the API expects that thought_signature to still be present in the conversation history. If it's missing → 400 Bad Request.

What the SDK sends — before vs what Gemini expects

What SDK sends (broken)
{ "contents": [ { "role": "user", "parts": [{ "text": "I go to school..." }] }, { "role": "model", "parts": [{ "functionCall": { "name": "save_english_mistake", "args": { ... } // thought_signature missing! } }] }, { "role": "user", "parts": [{ "functionResponse": { "name": "save_english_mistake", "response": { "result": "saved" } } }] } ] }
What Gemini expects (correct)
{ "contents": [ { "role": "user", "parts": [{ "text": "I go to school..." }] }, { "role": "model", "parts": [{ "functionCall": { "name": "save_english_mistake", "args": { ... }, "thought_signature": "AaBbCc..." // ← must be preserved! } }] }, { "role": "user", "parts": [{ "functionResponse": { "name": "save_english_mistake", "response": { "result": "saved" } } }] } ] }

The fix

Solution — disable thinking mode
this.model = client.getGenerativeModel({ model: 'gemini-3.5-flash', systemInstruction: '...', + generationConfig: { + thinkingConfig: { thinkingBudget: 0 } + } as never, // type not in SDK yet })

Setting thinkingBudget: 0 tells Gemini not to use thinking mode. No thinking → no thought_signature → the follow-up request is accepted and the agentic loop works.

Tradeoff of the current fix

What we gain

The agentic loop works. Gemini generates a real text response after calling save_english_mistake. No more empty bubbles or 400 errors.

What we give up

Deep internal reasoning before answering. For everyday chat this is negligible — the model still answers correctly, just without the extra scratch work.

Future — can we use the loop WITH thinking enabled?

Yes — but you have to bypass the SDK's built-in chat history and manage history yourself. The SDK's ChatSession is the only thing that loses thought_signature. The raw API is fine with it, so if you build the request manually you get both thinking and the loop.

How — SDK chat vs manual history

Current approach (SDK chat) — drops signature
// SDK manages history internally // but strips thought_signature const chat = model.startChat({ history }) // Turn 1 const r1 = await chat .sendMessageStream(userMessage) // Turn 2 — SDK history is broken here // ↑ thought_signature already lost const r2 = await chat .sendMessageStream([ { functionResponse: ... } ]) // → 400 Bad Request
Manual history — preserves signature
// You own the history array // Raw parts keep every field const history: unknown[] = [ ...previousMessages, { role: 'user', parts: [{ text: userMessage }] }, ] // Turn 1 — low-level API const r1 = await model .generateContentStream( { contents: history } as never ) // Collect raw parts (thought_signature intact) const modelParts: unknown[] = [] for await (const chunk of r1.stream) { for (const part of (chunk as any).candidates ?.[0]?.content?.parts ?? []) { modelParts.push(part) // ← raw, keeps sig } } // Append model turn with signature preserved history.push( { role: 'model', parts: modelParts } ) // Append function response turn history.push({ role: 'user', parts: [{ functionResponse: { name: 'save_english_mistake', response: { result: 'saved' } } }] }) // Turn 2 — history has thought_signature const r2 = await model .generateContentStream( { contents: history } as never ) // → works! text response streams normally
